# 项目概要

本项目是为了实现一个VRChat实时流式翻译的python软件，软件能够从 VRChat应用程序 或 系统麦克风 内实时读取音频流，把翻译后的文字（或多种翻译后的文字）通过VRChat的OSC功能发送出去或呈现在命令行里

# 项目运行逻辑

整个项目分为以下几个阶段：1. 从指定的音频流（VRChat 或 麦克风）读取音频信息 -> 2. 调用翻译接口，获取到翻译结果（可以是流式音频翻译接口，或者是传统文字识别+机器翻译接口） -> 3. 把翻译后的文字进行消费（比如通过VRChat的OSC功能发送出去，或者显示在命令行里）

注意本项目需要实现管道信息传递思想，也就是可以实现多个音频流以及对应的翻译以及文字消费功能。一个多消息管道同时消费的例子如下

VRChat音频（获取到英文音频）-> 调用服务商A的流式翻译接口 -> 把翻译后的文字显示在屏幕上（消费信息）
同时，电脑麦克风（获取到中文音频）-> 调用服务商B的语音转文字接口 -> 调用服务商C的文字翻译接口 -> 把翻译后的文字通过OSC发送到游戏内文字框显示（消费信息）

这意味着，每一个音频流/翻译文字可以视为一个信息包，这些信息包由最开始的模块产生，并沿着pipeline流动，逐个模块进行处理，每个模块会逐个消费它要处理的消息包，并生产新的消息包向后继模块传递。一个信息包可以转发至多个后续模块，而非只能单条链路的传递。

## 音频流获取

这个阶段需要从指定的音频流读取音频信息，你可以使用比较先进的 soundcard库的方法（可以reference的代码我放在 refer_resources/ 文件夹下了），实现监听VRChat程序的音频以及麦克风音频。确保这些音频后续能够被正常用于流式音频处理。你需要使用多线程以及通讯队列技术来确保音频流可以持续被获取，而不会因为其他流程的运行而被中断。

## 翻译接口

这个阶段需要根据 音频流，最终得到翻译后的文字。如果是实时音频流的流式翻译，那么翻译后的文字按照特定的管道逐个传递出去；如果是先进行语音转文字，再进行机器翻译，那么翻译后的文字打包成一个句子并依次传递向下一层级。

## 文字消费

最终得到的翻译文字（或者原文）可以通过多种方式进行消费，比如显示在终端里/通过VRC的OSC发送进游戏聊天框等。

# 项目模块搭建指示

每个阶段的这些功能都应该进行模块化，并像插件一样插进信息处理管道中对信息进行 消费/生产。
阶段一，二，三各自有一个父class，定义了这个阶段的模块的基本行为，以及祖父class作为所有模块的共通模板。

## 关于信息包流动

可以借鉴互联网TCP包以及路由的相关概念。将每个信息打包，包含这个信息的所有属性。信息随着管道顺着多个路由（模块）进行传递，不同于互联网包的是，这个项目里的每个包在生成的时候就决定了它要依次经过哪些模块，后续的模块进行包处理的时候，通过新增字典的 键值对 来为这个包添加这个阶段的模块所处理的信息（每个阶段的模块们要用到哪些关键的 键的名字 是在上面我们说的的父class里定义好了的，进行统一）

## 关于配置文件

插件思想搭建的项目，应当有一个配置文件，定义所有信息管道，以及每个信息管道内的模块运行所需信息（比如具体使用哪一个麦克风，比如调用api所需的base url和密钥是什么。）这个配置文件我的初步思想是使用json去进行定义，你需要尽可能使其更规范化，易于修改。

## 关于项目GUI

项目使用 core 核心模型，这意味着完成上述核心功能不需要GUI的参与。但是为了用户的易用性，需要有一个GUI来进行参数的更改，功能的开关。你可以使用 niceGUI 库进行一个网页页面的搭建，用户可以在上面进行配置文件的保存，服务的重启，阅读翻译后的文字（GUI页面也可以是文字消费的通道之一）等常见的功能。

## 关于兼容性

本项目只需要考虑 windows 平台上的正常运行
